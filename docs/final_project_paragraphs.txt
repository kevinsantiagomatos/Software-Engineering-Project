0027	[Heading1]	Introduction
0028	[]	This project presents the Paoli Group focused on improving the efficiency and effectiveness of the employee onboarding process. Paoli Group, a Puerto Rican technology firm recognized for delivering innovative software solutions, faced challenges with their hiring and onboarding system. The process of integrating new employees was inconsistent, time-consuming, and heavily reliant on manual documentation through platforms like OneDrive. This created delays of up to two weeks or more, reduced productivity, and left new hires unclear about their roles and responsibilities.
0030	[]	To address these issues, the project proposed the development of a centralized platform designed to streamline onboarding. The solution includes creating employee profiles, combining documentation in one place, automating task assignments, and enabling efficient training and follow-up. By replacing manual processes with a structured system, the platform ensures quicker integration of employees, improved clarity in roles, and better management of sensitive information.
0032	[]	The results of this project highlight how a customized onboarding solution can reduce inefficiencies, strengthen employee adaptation, and support Paoli Group’s commitment to quality, innovation, and excellence. Thus, this initiative positions the company to continue thriving in a competitive digital landscape while honoring its mission of fostering innovation and pride for Puerto Rico.
0035	[Heading1]	Progress Report
0036	[]	During this reporting period, the team completed a substantial set of functional and interface improvements to the Paoli Group onboarding platform, with emphasis on document management, workflow integrity, role-based behavior, and administrative review operations. The system has progressed from a prototype-like state into a more reliable and operationally aligned product, particularly in areas that directly affect HR and compliance execution.
0037	[]	On the backend, we strengthened the request and authorization flow to better separate browser navigation behavior from API behavior. In practical terms, this resolved cases where API calls were receiving HTML redirect content instead of proper API responses, which had been causing frontend instability. We also improved document delivery behavior by formalizing support for view and download URLs, enabling frontend modules to access files in a consistent and secure manner. Policy acknowledgment logic was updated so signatures are now tied to authenticated session identity rather than arbitrary user-provided email inputs, reducing risk of incorrect or spoofed submissions. Progress tracking was also corrected and standardized so profile and dashboard progress values now align with backend-calculated metrics.
0038	[]	A major logic refinement in this cycle was role-based requirement handling. Specifically, “Registro de Comerciante” was updated to apply only to contractor workflows and no longer appears as a required item for employee workflows. This change was implemented in both backend requirement computation and frontend rendering behavior, so requirement counts, missing-document indicators, and progress percentages now reflect the appropriate business rule by role. This directly addressed earlier mismatches where users could appear blocked by requirements not applicable to their employment type.
0040	[]	On the frontend, the document submission and review experience were significantly improved. The upload page now supports clearer per-document actions and cleaner requirement presentation, while the HR/admin review view now includes direct file viewing and downloading in-context. The review queue behavior was also adjusted to display pending-review documents as a true operational queue, so items disappear from that queue after approval or rejection as expected in standard review workflows. Additionally, several dashboard and profile usability issues were corrected, including progress display consistency and interface elements that previously caused layout instability under interaction.
0041	[]	We also completed a set of UX refinements to improve presentation quality and consistency across modules. These included structured hero headers for policy and training pages, clearer action affordances for dashboard cards, elimination of redundant navigation controls, and an under-construction placeholder for incomplete modules to prevent dead-end navigation. Cross-browser behavior, including Safari-specific failure patterns, was further stabilized through improved response handling and client-side parsing safeguards.
0042	[]	Regarding stakeholder engagement, the team held two meetings with Paoli Group during this period. The first meeting did not succeed as a demonstration due to technical execution issues, including failure to run the application correctly in-session. The second meeting was successful; the platform was demonstrated with the latest improvements, and the client expressed positive feedback regarding progress and implementation direction. During that session, the remaining scope was reviewed in detail, including completion priorities such as SSO integration, finalization of administrative capabilities, and additional polish for production readiness.
0043	[]	At this stage, the project is materially closer to completion, with core onboarding flows operational and significantly improved in reliability and business-rule compliance. The next phase will focus on remaining enterprise-level requirements, particularly SSO, final admin-side maturity, and end-to-end validation in preparation for final delivery.

0198	[Heading1]	Benchmarking (Previous Solutions)
0200	[]	To understand how our onboarding HR software should perform, we compared other existing software systems in a process called benchmarking. This process helped us identify the main strengths, weaknesses, and unique features of our system that should have in relation to other popular HR tools like BambooHR and Workday. By analyzing what makes this software great, we can get a better understanding of how the onboarding process functions and what challenges we might face in the process of developing ours for Paoli group. We found the top software used for carrying out this process, these are BambooHR, Workday, and Jira. However, these platforms differ in purpose and scale.
0201	[]	BambooHR is one of the most widely used HR systems for small and medium sized organizations. It is known for its clean, intuitive interface and straightforward setup, focusing on making onboarding accessible to non-technical users. The platform simplifies document management, task assignment, and communication between HR and new hires. It automates reminders and approvals, reducing manual follow-ups. However, its main limitation is its lack of deep customization. Organizations with unique workflows or specific compliance requirements often need to adapt their process to fit BambooHR’s structure. It also relies on a model based on subscription, which becomes expensive as the number of users grows, making it less viable for companies seeking a long-term and self-managed solution.
0202	[]	Workday, on the other hand, is designed for large enterprises that require a comprehensive collection of tools for recruitment, onboarding, payroll, benefits, and analytics. It excels in automation, data integration, and compliance management. Workday’s ability to bring together employee data across departments makes it an industry standard for operations in a corporate scale. However, this power comes at a cost, its implementation requires significant technical expertise, months of configuration, and high licensing expenses. The complexity of its interface can also make it overwhelming for smaller teams or organizations without a dedicated HR technology department. Despite its challenges, Workday sets a benchmark for what a fully integrated HR ecosystem can achieve when scalability and compliance are priorities.
0203	[]	Lastly, Jira, while not specifically an HR system, is widely used for project and workflow management due to its flexibility and customization options. Its task-tracking features can serve as a reference for structuring onboarding workflows efficiently, even though it is not open source and requires technical setup.
0204	[]	By comparing these tools, we were able to identify valuable elements, such as automation, ease of use, and task management, that can help us guide the development to one more efficient and user-centered onboarding system for Paoli Group.

0218	[Heading1]	Proposed Solution to the Problem
0220	[]	The current onboarding process at Paoli Group is inconsistent, heavily manual, and distributed across multiple platforms, causing delays, repeated work, and confusion for both new hires and HR staff. Documentation is stored across tools such as OneDrive, shared drives, and email, making it difficult to track progress, verify documents, or manage training efficiently. New employees often lack clarity regarding their responsibilities, while HR, IT, Compliance, and Project Managers struggle to coordinate tasks and approvals due to the absence of a unified system. These inefficiencies result in onboarding times of up to two weeks or more, inconsistent training, and unnecessary workload across departments. Therefore, a centralized solution is required to automate, standardize, and streamline the full onboarding cycle.
0221	[]	The proposed solution is the development of a centralized Digital Onboarding Management System (DOMS) that consolidates all onboarding workflows, documents, tasks, and communication into a single platform. Functionally, the system supports every activity modeled in the use cases: new hires can create accounts, upload documents securely, complete self-guided training modules, sign policies digitally, and track the status of their onboarding. HR personnel can validate documents, review submissions, manage profiles, approve steps, and monitor progress through dashboards. IT administrators configure access to tools such as email, Microsoft accounts, Slack, Jira, and SharePoint, ensuring that every hire receives proper credentials. Compliance officers review legal forms, verify acknowledgments, and record approvals within the system. Project Managers can assign training tasks, define project responsibilities, and verify readiness before onboarding is marked complete. The system replaces manual email exchanges, reduces training inconsistency, and ensures each stakeholder performs their tasks in a structured, traceable, and standardized workflow.
0222	[]	To complement the functional solution, DOMS incorporates essential non-functional requirements that ensure reliability, security, and scalability. The platform enforces data encryption for documents and personal information, ensuring compliance with privacy and labor regulations. The user interface is designed to be intuitive and requires minimal training, supporting usability for employees with varying levels of technical experience. System performance ensures quick response times even when multiple users interact simultaneously, while reliability targets high availability to prevent workflow interruptions. Audit logs and role-based access control provide accountability and controlled access to sensitive information. As Paoli Group grows, the system architecture supports scalability for additional employees, departments, and modules. These non-functional attributes guarantee that the platform not only functions correctly but also meet the quality standards required for a secure and modern digital onboarding environment.
0226	[Heading1]	Stakeholders
0228	[]	The primary stakeholders involved in the onboarding process include new employees, HR personnel, project managers, the IT administrator, and the compliance officer. In the current onboarding process, new employees must manually submit all required documents through email and often experience delays, uncertainty, and limited visibility regarding their onboarding status. Their role is important because they initiate the process, and any confusion or incomplete submissions immediately affect every other department. HR staff manually review, validate, and store these documents across different shared drives, which increases administrative workload and leads to inconsistencies or missing records. HR is crucial because they serve as the central point of coordination, ensuring accuracy, completeness, and proper documentation for each new hire. Project managers currently rely on external tools such as Jira to assign tasks and training steps, requiring manual cross-referencing to confirm whether the employee has completed essential onboarding actions. Their importance lies in preparing new hires for project integration and ensuring they meet the expectations and skills required for their assigned roles. Meanwhile, the IT administrator handles account creation and access setup independently of HR, causing repeated communication and delays in granting technical access. IT plays a vital role because it provides the systems, tools, and credentials that new employees need to be productive. The compliance officer manually verifies required legal documents, policy acknowledgments, and certifications, without automated tracking or audit trials. Their importance is fundamental because they protect the company from legal, regulatory, and contractual risks related to document management and employee compliance. As a result, the current onboarding process is inconsistent, slow, and heavily dependent on manual steps that make it difficult to standardize and track effectively.
0231	[]	In the proposed system, each stakeholder’s role becomes more structured, transparent, and aligned with a centralized digital onboarding workflow. New employees will upload required documents, complete assigned tasks, review policies, and track their progress directly through the platform. Their importance increases because the system empowers them to complete onboarding more independently and reduces delays caused by miscommunication or uncertainty. HR personnel will verify documents, approve onboarding stages, monitor progress, and manage employee information through automated workflows and standardized checklists. HR’s importance is reinforced because they ensure consistency, accuracy, and compliance at every step of the onboarding lifecycle. Project managers will assign training tasks, onboarding objectives, and project specific requirements directly through the platform and will have real time visibility of each hire readiness. Their importance lies in preparing new employees to contribute effectively and ensuring they are properly integrated into team workflows. IT administrators will manage account provisioning, access permissions, device setup, and license assignments using automated notifications and predefined configuration steps. IT’s importance comes from guaranteeing that employees receive timely, secure, and functional access to all technological resources. The compliance officer will review policy acknowledgments, signed documents, legal records, and retention requirements using digital signatures, audit logs, and compliance checklists built into the system. Their importance is essential because they ensure that the organization meets all regulatory obligations, prevents legal exposure, and maintains proper documentation. By redefining and integrating the responsibilities of each stakeholder within a unified platform, the onboarding process becomes more efficient, secure, transparent, and aligned with organizational best practices directly addressing the issues present in the current workflow.

0236	[]	For this project, we decided on the SCRUM development model due to it providing flexibility and feedback alongside the development of the project. Since we are building an HR onboarding system, SCRUM allows us to work in small components at a time and be able to test and receive feedback in an efficient and fast way. As well as being easier to organize the team and assign tasks in a productive manner. We will divide into 5 sprints, where in each we will develop a portion of the software be it database design and development, login information, document management and storage. Each sprint will consist of planning, development and feedback to ensure we can provide as much of what the client has requested to the best of our abilities. SCRUM also allows improvements to be made retroactively without compromising other aspects of the project. Although using SCRUM process will be time consuming as the frequent feedback process will take some time, as well as compromising the development schedule. However, this drawback will prove beneficial for the project since we will be acquiring the necessary feedback to create a better software.

0660	[]	4. Database Model
0661	[]	4.1 E-R Relational
0666	[]	Entities
0667	[]	4.2 New_hire Entity
0668	[]	New_hire entity stores the personal data for soon to be employees but are not yet activated as system users. This includes name, email, phone, date of birth, and complete address data, as well as job-related fields such as the department, role, assigned manager, and projected start date.
0670	[]	4.3 New Hire Attchment Entity
0671	[]	The New_Hire_attachment entity stores documents that have been uploaded for the new hire; it stores the original name, stored name, URL, document Id and type. It has a one to many relationship with New_hire, with hire_id being the foreign key. A single document belongs to one user, but a user can have many distinct documents.
0674	[]	4.4 UserEntity
0675	[]	The user entity stores the information of individuals are active accounts. This includes their email address, full name, hashed password, role, department, account status, profile image, and the timestamp indicating when their account was created. It has a one-to-many relationship with Task, as it has one owner and can have many employees. It also has a many-to-many relationship with Training_Module, implemented through the Training_Completion table. Additionally, the User entity links to Policy_Ack in another many-to-many relationship with Policy, capturing digital acknowledgments and compliance confirmations. The User entity also forms a one-to-many relationship with Document, records all the files uploaded by that user, as well as a one-to-many relationship with IT_Provision, which stores any IT resources, accounts, or equipment issued to the User. Finally, the User entity participates in a one-to-many relationship with Audit_Log, representing every action the user performs on the system for accountability and historical tracking.
0677	[]	4.5 Address Entity
0678	[]	It provides a structured way to record and identify each user’s full address, allowing other entities to link to it through the address_id field. This table connects to the User table, meaning it stores the address information that belongs to each user created.
0680	[]	4.6  Task Relation
0681	[]	The Task entity contains all onboarding-related tasks assigned to a user. Include an Id of the task, title, description, email of the supervisor, and who assigned it, as well as the category of a task. Also shows the dates of when it was created, updated and its due date.
0683	[]	4.7  Document relation
0684	[]	The Document table stores the documents submitted by the User, the stored name, the name and email of the uploader, and whether the document is accepted or rejected. A user can submit multiple documents, but each document is tied to a single User.
0688	[]	4.8 Training_completion entity
0689	[]	The Training_compleiton entity has which training module the User has completed. It includes the completion timestamp and the id from training_module as well as the email of the supervisor.
0691	[]	.9 Training_module entity
0692	[]	The training_module entity stores the information of the modules that are assigned to the user, such as the module Id, title, and description. It has a many to one relation with User as there can be many training modules assigned to one User.
0694	[]	4.10 It_provision entity
0695	[]	I-T provision entity houses the Id of the provision and when it was assigned to the User. It has a many to one relation with the user, since the User can have many provisions assigned.
0698	[]	4.11 Policy_ack entity
0699	[]	Policy_ack houses the policy that was acknowledged to the users, as well as the user email, acknowledgement, signature and timestamp of the signature. This entity is a one to many relationship with the user as a user can acknowledge many policies as part of the onboarding process.
0701	[]	4.12 Audit log relation
0702	[]	The audit_log entity stores the actions made by the user. It includes the type of action done, the user_id and timestamp of when it was done.

0830	[Heading1]	Ethics
0832	[]	The ethical implications of developing the Digital Onboarding Management System (DOMS) for Paoli Group LLC were evaluated throughout the project using the ACM/IEEE Software Engineering Code of Ethics. Because DOMS processes sensitive legal and personal data, the team analyzed each ethical principle alongside the specific risks related to onboarding, document handling, access configuration, and compliance processes. The following sections map each principle to the real risks involved in DOMS and explain how the project mitigated them through design, decision-making, and responsible engineering behavior.
0834	[Heading3]	1. Public Interest
0835	[]	DOMS impacts public interest by protecting individuals’ sensitive information while supporting efficient organizational operations. The primary risk in this category was exposing private documents or allowing unauthorized access due to poor implementation. To mitigate this, the team implemented encryption, access control, audit trails, and restricted visibility based on role permissions.

0970	[]	The onboarding begins with the "Registered" state, initiated by HR when a new employee profile is created within DOMS. Once this stage is completed, the state transitions again to "Project Assignment," where managers assign tasks, mentors, and onboarding objectives. Finally, the employee enters the "Training and Orientation" state, during which they complete assigned training modules and tasks. The process culminates in the "Completed" state once all onboarding steps are successfully fulfilled.
0977	[Heading1]	Architectural Design
0978	[]	Architectural Approach
0979	[]	The architectural design of the Digital Onboarding Management System (DOMS) is based on a hybrid approach that combines Layered Architecture, Model-View-Controller (MVC), and the Client–Server model. This structure improves modularity, maintainability, and scalability by separating data management, business logic, and user interface components while supporting a distributed and secure operational model.
0980	[]	At the core of the system is the Layered Architecture, which organizes DOMS into four primary layers:
0981	[]	Presentation Layer: This layer contains the front-end interface used by New Hires, HR, IT, Compliance, and Project Managers. It includes dashboards, document upload screens, training pages, and progress views, ensuring user accessibility and clarity.
0982	[]	Application (Business Logic) Layer: Responsible for implementing onboarding workflows such as document validation, IT access configuration, compliance reviews, and project assignment. This layer ensures the system enforces the correct rules and processes at each stage.
0983	[]	Service/Integration Layer: Manages communication with external services such as digital signatures, email notifications, and identity validation. It prepares the system for future integrations and expansions.
0984	[]	Data Layer: Contains the relational database that stores user profiles, documents, onboarding statuses, approval logs, and event history. Centralizing the data ensures consistency, security, and auditability.
0987	[]	The system also utilizes the Model-View-Controller (MVC) pattern, which aligns with modern web application development.
0988	[]	The Model represents all application data such as users, roles, documents, tasks, and onboarding progress.
0989	[]	The View provides the interface components that present information to each stakeholder.
0990	[]	The Controller manages operations including login, document processing, task assignment, compliance approval, and workflow transitions.
0991	[]	Using MVC ensures that updates to the interface, database, or logic can occur independently without affecting other components, improving system flexibility.
0992	[]	Because DOMS operates as a web-based platform, it also incorporates a Client–Server Architecture. The client (web browser) handles user interactions, while the server processes the business rules, manages workflows, and communicates with the database. This model increases reliability, supports multiple concurrent users, and ensures all operations are performed securely and consistently.
